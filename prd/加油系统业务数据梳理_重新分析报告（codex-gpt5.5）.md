# 加油系统业务数据梳理重新分析报告

生成时间：2026-05-29  
分析对象：`prd/加油系统业务数据梳理.xlsx`

## 1. 结论

该文档整体是合理的，已经覆盖了机场加油系统从航班接入、订单、派工、任务执行、油单、监控报警到系统管理的大部分核心业务。相比普通需求清单，它更接近“业务流程 + 数据对象 + 系统模块”的综合设计稿，业务理解较扎实。

但它还不适合作为直接开发的数据模型或详细设计依据。主要原因是：数据结构定义过于 Excel 化，缺少标准 ER 模型和字段级约束；部分关键流程以图片形式嵌入，难以版本管理和自动校验；非功能需求、审计、接口契约、结算财务、质量管理等落地要素仍不完整。

综合评价：**7.5 / 10**。  
定位判断：**业务设计较完整，工程化设计不足，需要补充详细设计后再进入开发实施。**

## 3. 文档结构概览

工作簿共 27 个 Sheet，主要分为五类：

1. 总体架构类：总体业务结构、加油管理模块梳理。
2. 核心流程类：航班管理、资源管理、油料参数、派工、任务执行、任务管理、油单管理。
3. 扩展业务类：SAF 物料、SAF 订单、SAF 加注计划、WEBGIS、压差、过滤器维保、一机一位、静电监测、车辆效能分析。
4. 支撑管理类：系统管理、航显接入、调度设置、短语管理、报警管理。
5. 数据治理类：数据结构定义、技术数据分类、技术+业务数据分类、加油流程数据链路。

其中 `数据结构的定义` 包含约 4673 个非空单元格，是全表最重的数据定义 Sheet。

## 4. 设计合理之处

### 4.1 总体业务结构基本完整

总体结构图把系统拆成接口管理、订单管理、派工管理、资源管理、油料参数管理、任务管理、监控报警、辅助加油管理、设备设施管理、人员管理、系统管理和数据分析。这个分层是合理的，基本覆盖了一个机场航油调度系统的主干能力。

尤其是主流程关系比较清楚：

- ACDM / 航油客户应用系统进入订单和航班侧。
- 订单进入派工管理。
- 派工依赖资源管理和油料参数。
- 加油员执行任务后上传任务进度并生成、打印、上传油单。
- 油单再进入云平台、标准化和后续 ERP / 收款 / 支付 / 开票 / 核销链路。

这说明文档不是孤立功能罗列，而是试图表达业务闭环。

### 4.2 核心加油业务链路完整

端到端加油流程图覆盖了以下关键阶段：

- 航显对接 / 手工航班。
- 订单同步。
- 创建或更新航班信息。
- 任务派发。
- 加油开始与结束。
- PAD 任务信息与油单生成。
- 油单签字、打印、PDF 生成。
- 油单上传前端后台。
- 发送云平台。
- 油单标准化。
- 航司确认。
- 结算及 ERP 对接。

这条链路是合理的，也体现出系统已经考虑到现场 PAD、前端后台、云平台和结算系统之间的数据流转。

### 4.3 派工规则贴近真实业务

派工流程对调度判断条件描述较细，包括：

- 飞机已落地、上轮挡、确认加油量。
- 按加油量和预计起飞时间划分 60 / 90 / 120 分钟派工窗口。
- 快速过站、高峰保障、提前加油。
- 机位路线、滑行道等待影响。
- 特殊机型、加油员资质、专用车辆。
- 工作量均衡、距离成本、疲劳和天气因素。

这些规则具有现场业务价值，为后续自动派工或 AI 派工提供了较好的规则基础。

### 4.4 SAF 设计比较前瞻

文档将 SAF 拆成物料维护、订单管理和加注计划管理三个流程，而不是简单加几个字段。这是合理的，因为 SAF 涉及物料类型、工艺、掺混比例、保税类型、订单关联、计划匹配、航班标注和油单拆分。

较好的设计点包括：

- SAF 计划单支持机场、生效日期、客户代码、航司、航段、航班号、机号等匹配条件。
- 支持规则优先级。
- 计划单“向后生效”，避免历史航班被批量误改。
- 油单侧支持 SBC 与 3 号喷气燃料数量拆分。

### 4.5 IoT 与现场设备场景考虑充分

文档覆盖了车辆定位、WEBGIS、过滤器压差、静电监测、OBD 效能分析、PAD 离线缓存等现场设备数据。尤其压差电子化流程中，已经明确：

- 实时压差不持久化，仅用于实时显示。
- 最大流速压差随任务保存。
- PAD 离线时缓存最大流速压差，实时数据可丢弃。
- 异常压差、压差变动、压差计击穿有独立报警。

这种设计对实时数据、任务结果数据和告警数据做了初步区分，方向正确。

### 4.6 数据分类意识较好

数据分类图从技术角度区分了：

- 元数据。
- 内部数据 / 外部数据。
- 结构化数据 / 非结构化数据。
- 报表数据、事务数据、主数据、基础数据、观测数据、规则数据。

业务+技术分类图进一步把航班报表、加油量报表、事务类、观测类、规则类、主数据、基础数据做了映射。这说明文档已经开始考虑数据治理，而不只是功能设计。

## 5. 主要问题

### 5.1 数据模型不够工程化

`数据结构的定义` 内容量很大，但表达方式是横向铺开的 Excel 字段矩阵。它能用于业务梳理，但不适合作为数据库设计直接依据。

当前缺少以下内容：

- 实体之间的 ER 关系。
- 主键、外键、唯一约束。
- 字段类型、长度、精度、单位。
- 是否必填、默认值、枚举值。
- 生命周期状态机。
- 索引和查询场景。
- 数据保留期限。
- 敏感字段和脱敏策略。

建议把 `数据结构的定义` 拆成标准数据字典，至少包含：实体名、表名、字段名、字段说明、类型、长度、单位、是否必填、枚举、来源、去向、主外键、索引建议、数据分类。

### 5.2 引用体系有价值，但存在维护风险

文档大量使用 `REF x.x.x` 交叉引用。优点是可以追踪字段来源，缺点是手工维护成本高。

本次检查发现明显错误引用：

- `ref55.1.3.5`，疑似应为 `ref5.1.3.5`。
- `ref52.5.2.6`，疑似应为 `ref5.2.2.6` 或 `ref2.5.2.6`，需结合上下文确认。

建议后续用结构化数据字典或脚本校验 REF 是否存在，避免需求变更后引用失效。

### 5.3 图片化 Sheet 不利于协作和版本管理

总体业务结构、数据分类和加油流程都以图片嵌入 Excel。视觉上直观，但会带来几个问题：

- 无法搜索图中文字。
- 无法自动比对版本差异。
- 无法被脚本读取和校验。
- 后续修改依赖原始绘图文件，Excel 内图片很难维护。

建议把这些图转为 Mermaid、Draw.io 源文件或 Visio 源文件，并在报告中保留图片导出版本。

### 5.4 非功能需求不足

文档对业务操作写得多，但对系统工程指标写得少。当前缺少：

- 航班动态刷新延迟。
- 车辆定位上报频率和最大延迟。
- PAD 离线可用时长。
- 高峰期任务并发量。
- 油单上传重试策略。
- 消息幂等策略。
- 系统可用性目标。
- 灾备和降级方案。
- 审计日志和操作追溯要求。

这类内容不补齐，开发阶段容易出现“功能能跑，但生产不可控”的问题。

### 5.5 接口契约仍需细化

文档提到了 ACDM、航油客户应用系统、云平台、ERP、航显接入等外部系统，但缺少正式接口契约。

建议补充：

- 接口协议：REST、MQ、文件、数据库同步等。
- 消息结构。
- 频率和触发条件。
- 幂等键。
- 错误码。
- 重试与补偿。
- 对账机制。
- 数据归属和冲突处理规则。

特别是航班更新、订单更新、油单上传、航司确认这些链路，需要明确“谁是数据主源”。

### 5.6 财务、合同和质量管理仍偏弱

总体流程图已经把油单标准化后续连接到 ERP、收款、支付、开票、核销，但正文 Sheet 中对这些内容展开不足。

建议补充独立模块或至少补充流程：

- 客户 / 航司合同。
- 价格维护。
- 结算规则。
- 欠费规则。
- 账单生成。
- 对账与差异处理。
- 开票和核销。
- 油品质量检测记录。
- 质量异常追溯。

这些内容和油单、订单、SAF、保税属性强相关，后续不能只停留在“发送 ERP”层面。

### 5.7 AI 赋能停留在概念层

文档里多处保留了 AI 赋能行，但多数为空或描述较粗。当前还不能支撑 AI 功能开发。

如果要做 AI 派工助手，至少需要定义：

- 输入特征：航班、机位、油量、资源位置、资质、车辆类型、天气、历史耗时等。
- 输出结果：候选加油员、车辆、预计到位时间、风险提示。
- 解释信息：为什么推荐该派工。
- 人工覆盖机制。
- 置信度和兜底策略。
- 训练数据来源和效果评估指标。

否则 AI 容易变成页面上的辅助口号。

## 6. 模块级评价

| 模块 | 评价 |
|---|---|
| 航班管理 | 功能细，包含视图、过滤、搜索、追踪、修改、移动复制、短语、导出、排序、分组、闹钟。合理。 |
| 资源管理 | 能支持调度决策，但资源状态模型还需细化，例如在线、空闲、执行中、离线、故障、禁用等状态转换。 |
| 油料参数 | 有公式和校验规则，设计较扎实；但需要明确参数版本、生效范围、回滚和审计。 |
| 派工管理 | 业务规则丰富，是文档最有价值部分之一；建议进一步状态机化和规则参数化。 |
| 任务执行 | PAD 侧流程完整，覆盖登录、任务、到位、油量确认、开始结束、签字、打印、上传。合理。 |
| 任务管理 | 能满足监控和人工干预，但需要严格定义哪些状态允许人工改、是否需要审批和审计。 |
| 油单管理 | 覆盖查询、审核、修改、上传、下载、抽油单；需强化版本管理、作废规则和财务联动。 |
| SAF | 设计较完整，建议补充 SAF 计划、订单、航班标注、油单拆分之间的一致性校验。 |
| WEBGIS | 场景完整，但地图数据、GPS 数据和报警规则需要接口与性能指标支撑。 |
| 压差 / 静电 / 过滤器 | 现场设备管理设计较细，是文档亮点；需补充设备故障、数据缺失和误报警处理。 |
| 系统管理 | RBAC 思路正确，但字段级权限、审计日志、令牌过期和多端登录策略仍需补充。 |
| 数据分析 | 目前更多是分类和方向，缺少指标口径、报表清单和数据集市设计。 |

## 7. 建议的后续整改优先级

### P0：进入开发前必须补齐

1. 把加油任务、油单、航班、资源、油料参数、SAF 计划、SAF 订单整理成标准 ER 图。
2. 把 `数据结构的定义` 拆成标准字段表。
3. 明确任务状态机和油单状态机。
4. 明确航显、订单、云平台、ERP 的接口契约。
5. 修正 REF 错误并建立引用校验机制。

### P1：上线前必须补齐

1. 补充操作审计日志。
2. 补充异常补偿：航班更新失败、油单上传失败、PAD 离线、云平台返回失败。
3. 补充性能指标和高峰保障容量。
4. 补充结算、开票、核销、欠费规则。
5. 补充质量检测和质量异常追溯。

### P2：优化增强

1. 将图片化流程图改为可维护源文件。
2. 建立数据分类标签：主数据、事务数据、观测数据、规则数据、报表数据。
3. 定义 AI 派工助手的数据输入、输出、解释和评估指标。
4. 建立报表指标口径库。

## 8. 最终判断

该设计的业务方向是合理的，尤其在派工、任务执行、油单、SAF、压差、静电、WEBGIS 等机场航油特色场景上体现了较强的业务经验。

主要短板不在“是否懂业务”，而在“是否能让研发团队稳定落地”。当前文档仍偏业务梳理，缺少详细设计中的数据约束、接口契约、状态机、异常补偿、审计和性能指标。

建议把它作为需求与业务设计基线继续使用，但不要直接作为数据库和接口开发依据。下一步应产出三类补充文档：

1. 数据模型详细设计。
2. 接口与消息契约设计。
3. 状态机与异常补偿设计。
