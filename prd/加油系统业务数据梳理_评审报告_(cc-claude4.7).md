## 一、文档定位的再确认

读完文档后，最关键的一点是先认清它是什么。

**它不是**：
- 一份从零开始的新系统设计文档
- 一份技术规格说明（SRS）或数据库设计说明
- 详细的需求规约（无验收标准、无优先级、无版本规划）

**它是**：
- 一份"业务-产品-数据"三视角的**业务梳理矩阵**
- 服务于存量浦东加油系统在 **SAF（可持续航空燃料）业务上线** 节点上的整体业务盘点
- 兼具"老业务回顾"+"新业务（SAF、过滤器压差电子化、静电监测、一机一位）规划"的混合体

> **定位错配是一切评审分歧的根源**——用 SRS 的标准去要求一份业务梳理表，会得出"非常不合理"的结论；用业务盘点矩阵的标准去看，则结论是"基本合理但需补强"。

## 三、做得好的部分（无需改动的设计）

### 3.1 业务矩阵的二维切分非常专业

横轴：业务子流程（如"航班监控、强制下线、发送短语..."）
纵轴：运营角色 + 产品职责 + AI 能力 + 数据结构 REF

这种排版的价值是：**在一张表里同时锁定"谁来做、做什么、用什么数据"**，避免需求评审会议陷入"流程归流程、数据归数据"的脱节状态。这是行业内成熟做法（与 IBM Component Business Map、阿里业务中台业务域梳理路数一致）。

### 3.2 SAF 业务规则的严谨度超出预期

文档中 SAF 油单拆分给出了完整的边界条件处理：

```
SBC重量 = 总加油重量 × 加注掺混比例（四舍五入到整千克）
SBC升数 = SBC重量 / 密度（四舍五入到整升）
3号喷气燃料重量 = 总加油重量 - SAF重量
3号喷气燃料升数 = 3号喷气燃料重量 / 密度（四舍五入到整升）
```

且明确告知"拆分后的 SAF 升数 + 3号喷气燃料升数 ≠ 油单总升数 是正常的"——这是会计/计量领域常见陷阱，文档点出来就避免了下游开发踩坑。

更重要的是对极端情况（SBC 重量四舍五入为 0）给出 **方式一/方式二两种实现路径**，由前端配置切换。这种"保留选择权"的写法对存量系统改造非常友好。

### 3.3 SAF 加注计划匹配的"向后生效"规则

> "SAF 加注计划单（包括计划单的创建、修改或取消）的生效规则为'向后生效'，即仅对在 SAF 加注计划创建/修改/取消后再接入或更新的航班有效"

这条规则看似简单，实际上避免了一个分布式系统典型陷阱：**如果改成"向前回溯生效"，会引发已派工任务、已生成油单的级联回滚**。明确"向后生效"+"由航班接入/更新触发匹配"的事件驱动模型，工程上正确。

### 3.4 油料参数密度校验公式可直接落代码

```
计算密度 = (20 - 输入温度) × 0.00073 + 标准密度
通过条件：计算密度 × (1-0.0005) ≤ 输入密度 ≤ 计算密度 × (1+0.0005)
```

这是石化行业 ASTM D1250 体积温度修正的简化版，公式来源可靠，可作为唯一真值源（Single Source of Truth）直接被前端、PAD、后台三处复用。

### 3.5 派工规则覆盖维度合理

7 条派工规则覆盖了 **时间窗、资质匹配、地井条件、距离优化、负载均衡、人员健康/疲劳、天气** 七个维度。从运筹学角度看，这是典型的多目标带约束的指派/路径问题（OR-Tools 可解），文档把约束列清楚了，下游可直接据此选算法（文档也给了候选：匈牙利、CVRP、神经网络、大模型）。

### 3.6 压差电子化、静电监测、过滤器维保 — 设备物联化思路正确

这三个流程都遵循了 **"航油一体机采集 → 后台转换计算 → 缓存+持久化 → 报警+查询"** 的统一物联骨架。统一骨架的好处：未来新增传感器（如温度、振动）可按相同套路扩展，不需重新设计。

## 四、不合理或风险点（按严重度排序）

### 🔴 【高】4.1 缺少与存量代码/表结构的映射

文档大量使用 `REF 2.5.1`、`REF 5.1.2` 等内部引用，但 **没有任何一处映射到现有数据库（`db_oil_pvg`，200+ 表）和模块（28 个）**。

后果：
- 开发拿到这份文档，第一件事仍然要去 `flight-oil`、`flightdispatch`、`terminalmanager-oil` 翻代码确认"航班数据"到底对应哪些表
- "航班变更日志（REF 2.5.4）"是新建表，还是复用 `t_flight_history`、`t_flight_change_log`？无答案
- 对 SAF 这种**全新业务**，必须明确"复用 `t_oil_bill` 加字段 vs 新建 `t_oil_bill_saf`"——文档回避了这个决策

**建议**：在 `数据结构的定义` Sheet 加一列"实现映射"，标注每个数据项对应的现有表名+字段名（或"待新建"）。这件事工作量不大，但对开发的价值是 10 倍级的。

### 🔴 【高】4.2 异常处理流程几乎全部缺失

矩阵全是"happy path"，关键异常路径未覆盖：

| 业务环节 | 文档未回答的关键问题 |
|---------|-------------------|
| 派工后航班取消 | 已派任务是自动取消、转下一航班、还是人工确认？ |
| 加油中加油员设备故障 | 任务如何转交？已记的油量数据如何归集？ |
| PAD 离线签单 | 离线生成的油单，与后台 SAF 标记冲突时谁优先？ |
| SAF 加注计划取消 | 已关联但未加油的航班，SAF 标记是否清除？（文档说会清除，但已派工的呢？） |
| 油料参数发布失败 | 已下发任务用旧参数还是阻塞派工？ |
| 网络抖动重复上报油单 | 幂等键是什么？（油单号？任务编号？） |

**关键漏洞**：第 4.2.1 项（航班取消已派工处理）和 4.2.2 项（PAD 离线冲突）是真实生产环境每天发生的场景，已存量系统肯定有对应代码，但本梳理文档没体现 —— 这意味着 SAF 业务上线时这些异常分支的 SAF 处理逻辑全是盲区。

### 🔴 【高】4.3 SAF 业务与油单/任务的事务边界没说清

SAF 业务横跨多个流程（订单管理 → 加注计划 → 航班标记 → 任务派发 → 油单生成 → 拆分上传 ERP），其中三处事务边界文档没说：

**问题 1**：SAF 加注计划匹配是 **航班接入时同步匹配**，还是 **异步事件**？
- 同步：航班接入接口性能受影响
- 异步：可能出现"航班已派工但 SAF 标记还没打上"的窗口

**问题 2**：SAF 标记打上后，**已生成的待加油任务是否回填 SAF 信息**？文档说"任务下发时携带 SAF 信息"，但任务可能在 SAF 标记前已下发。

**问题 3**：油单 SAF 拆分后的 SBC 数量与 3号喷气燃料数量，是 **生成油单时一次性计算并持久化**，还是 **每次查询/上传时实时计算**？两种实现的并发安全性与改密度后的可追溯性完全不同。

### 🟡 【中】4.4 "短语管理"与"报警管理"双轨制隐患

文档把 **工作短语** 和 **系统报警/提醒** 拆成两个独立流程，但二者在数据字典里强耦合：
- `1.8.9 短语消息类型` 包含了"系统消息、进程报告、情况反映、催办、变更提示、机位提醒、机场公告、警告..."等
- `1.8.4 报警类型` 列了 23 种报警

**问题**：一条"航班变更提醒"（`1.8.4.20`）到底走"短语通道"还是"报警通道"？两者都能"调度接收后弹小窗口显示"，UI/UX 容易冲突。
- 历史上，浦东系统的告警体系应该已经形成（`messagemanager` 模块就是干这个的）
- 本次新增的"压差报警/静电报警/一机一位提醒"等，是接入老报警体系，还是另起一套？文档未表态

**建议**：在 `报警管理流程` Sheet 增加一节"通道路由表"，明确每种事件走 PAD 弹窗 / 调度小窗 / 短语 / 邮件 / 短信 中的哪几个组合。

### 🟡 【中】4.5 "AI 赋能"列填得不够诚实

很多 Sheet 的"AI 赋能"列空着，但少数地方写了：
- "AI 派工助手"
- "基于语音的信息报送、查询"
- "AI 进行预防性报警"
- "错误油单 AI 识别"

这些是 **未来要做** 还是 **本期要做**？是 **本地小模型** 还是 **接 LLM API**？模型在哪里推理（PAD 上、后台、云）？没说。

风险：AI 列容易被解读为"产品承诺"，进入测试用例和验收范围，但实际工程预算和落地时间都未敲定。

**建议**：要么删掉 AI 列（如果是远期想法），要么补一行"AI 实施阶段"标注 P0/P1/P2 优先级。

### 🟡 【中】4.6 数据结构 Sheet 严重残缺

`数据结构的定义` Sheet 实际只完整列出了"5.x 监控/统计层"和"1.x 业务字典枚举层"，但中间最关键的 **2.x（基础数据）、3.x（业务实体）、4.x（业务规则）** 三大段都缺失。

举例：
- `航班数据 REF 2.5` 被 17 次引用，但 Sheet 中找不到 2.5 的字段清单
- `加油任务 REF 5.1` 在 5.1.2 给出了 24 个字段，但 `任务进程 REF 5.1.3` 只给了 5 个，对比 `任务监控` 业务的复杂度明显不足
- `派工黑名单规则 REF 3.1.8`、`派工白名单规则 REF 3.1.9` 在业务流程被引用，但数据结构 Sheet 中查不到定义

这种"被引用却未定义"的引用约 30+ 处，构成了文档自身的引用完整性漏洞。

**建议**：先用脚本扫一遍所有 `REF x.x.x` 引用，做一份"引用一致性检查报告"，定位所有"引用了但没定义"和"定义了但没引用"的项。

### 🟡 【中】4.7 与存量技术债务的协同策略缺位

`doc/readme.md` 明确列了存量系统的高风险技术债：
- Java 1.7、Spring 3.x/4.x、Dubbo 2.5.x 均已停止维护
- 存在 SQL 注入与明文密码

而本梳理文档新增了大量功能（SAF、压差监测、静电监测、一机一位、AI 派工...），**全部 imply 在老技术栈上新增**。这会进一步增加未来升级负担：
- SAF 油单拆分写到 `flight-oil` 还是 `oil-extend-sys`？后者本来是扩展系统，承担新业务更合理
- 新加的物联设备数据（压差、静电）走老 ActiveMQ 还是引入新的 IoT 接入层？

**建议**：业务梳理文档之外，应有一份配套的 **"实施承载决策表"**，回答每个新功能落到哪个模块、是否需要新模块、是否触发部分老模块升级。

### 🟢 【低】4.8 引用编号有少量笔误

## 五、结合存量系统的关键建议

存量系统的现实约束应改变本次梳理的输出方式。具体建议：

### 5.1 在文档中新增"实现承载"列

每个业务子流程加一列：

| 子流程 | ... | **实现承载模块** | **是否触发存量改造** |
|--------|-----|---------------|------------------|
| SAF 油单拆分 | ... | oil-extend-sys（新） | flight-oil 加字段 |
| 一机一位提醒 | ... | terminalmanager-oil | 新增 PAD 端能力 |
| 压差报警 | ... | messagemanager | 新报警类型注册 |

### 5.2 SAF 业务建议独立成卷

SAF 涉及的业务子流程已经超过 20 个（物料、订单、加注计划、航班标记、任务下发、油单拆分、ERP 上传、结算），建议从本表抽出来形成 **"SAF 业务专项设计"独立文档**，避免与老业务搅在一起。

### 5.3 增加"灰度上线"维度

存量系统有日处理数百架次航班的稳定运行实际，任何变更都需灰度。建议每个新功能加"灰度策略"标注：
- 按机场：浦东先上、其他后续
- 按航司：东航试点
- 按航班类型：内航/外航分批
- 按时段：夜间低峰试点

### 5.4 数据迁移与回滚预案

SAF 引入了多张新表（SAF 物料、加注计划、油单 SAF 信息），建议明确：
- 上线时是否需要历史航班数据回填 SAF 标记（默认应为否，配合"向后生效"规则）
- 老油单表新增 SAF 字段的 DDL 默认值
- 万一回滚，已生成的 SAF 油单怎么处理

## 附录：方法论说明

本评审采用了 **业务建模成熟度分级法（CBM Maturity Levels）**：
- L1 业务实体识别 → 文档达成 ✅
- L2 业务流程覆盖 → 文档达成 ✅
- L3 数据结构定义 → 部分达成 ⚠️
- L4 事务一致性与异常路径 → 未达成 ❌
- L5 工程落地承载 → 未达成 ❌

存量系统的业务梳理通常要求达到 L4；新业务（SAF）应直接达到 L5。本文档目前在 L2.5 左右，距离工程可用尚有 1–2 个迭代的补强空间。
